# ツギイチ StoreKit2 PRレビュー・チェックリスト（Devin用）
目的：サブスク + 消耗型IAP（AIパック）の実装が「購入できた」で終わらず、復元・解約・審査・不正・例外まで破綻しないことを確認する。

---

## 0. 前提（ツギイチの課金仕様）
- サブスク（月額）：`tsugiichi.pro.monthly`
- 消耗型（AI追加パック）：`tsugiichi.ai.pack300`
- Free：AI 10回 / 30日
- Pro：AI 300回 / 30日
- 消耗型：+300回（期限なし、月次枠→購入枠の順で消費）
- iOSアプリにOpenAIキーを埋め込まない（Proxy経由）

---

## 1. 購入処理（Purchase Flow）
- [ ] 商品一覧取得が非同期で安定（ローディング/失敗/再試行UIあり）
- [ ] 「購入」タップ → 購入中UI（連打防止）→ 成功/失敗/キャンセルを正しく区別
- [ ] `userCancelled` を「エラー扱い」でアラート連発しない（静かに戻す）
- [ ] 失敗時にアプリがクラッシュしない（ネットワーク不安定、Store不調）
- [ ] 購入完了後、すぐにUIが更新される（Pro解除・残回数表示など）

---

## 2. 取引の検証（Verification）
- [ ] `Transaction` の検証（StoreKit2の verification を必ず通す）
- [ ] 検証できない場合の扱いが明確（Pro付与しない / 消耗型クレジット加算しない）
- [ ] ログ/デバッグ情報が残っている（本番で個人情報は出さない）

---

## 3. 復元（Restore Purchases）
- [ ] Settingsに「購入を復元」導線がある
- [ ] 復元実行 → 成功/失敗/キャンセルのUIがある
- [ ] サブスク復元で isPro が正しく反映される
- [ ] 消耗型の“復元”の扱いが仕様どおり
  - 消耗型は基本「使い切り」なので、実装が二重加算にならないよう注意
  - ツギイチでは「購入分クレジット」はローカル保持＋Proxy制御（MVP）前提なので、二重付与を防ぐ実装が必要

---

## 4. サブスク状態監視（Subscription Status）
- [ ] `currentEntitlements` / `subscriptionStatus` で isPro 判定（キャッシュだけに依存しない）
- [ ] 解約/期限切れ/支払い失敗で isPro が落ちる（更新タイミングも確認）
- [ ] `Transaction.updates` を監視して状態を追従（アプリ起動中に状態が変わっても反映）
- [ ] オフライン時の扱いが破綻しない（最後に確認できた状態の扱いを決めている）

---

## 5. 消耗型（Consumable）の扱い
- [ ] `tsugiichi.ai.pack300` の購入で +300 が加算される
- [ ] 二重購入（二重タップ）で +600 になるなど仕様どおり（防ぐなら防ぐ）
- [ ] 購入後のクレジット表示が即反映
- [ ] 消費順序が仕様どおり（Free/Pro月次枠 → 購入枠）
- [ ] Proが切れても購入枠は残る

---

## 6. 30日ローリングウィンドウ（Free/Pro）
- [ ] Free枠：初回AI利用時刻を起点に30日でリセットされる
- [ ] Pro枠：Pro有効開始（or 更新）起点で30日枠が管理される
- [ ] 端末時刻変更に強い（少なくとも破綻しない）
  - 可能なら「サーバ時間/Proxy時間」を正とする（M11以降）
- [ ] ローカルだけの制御で無限化しない（M11でProxy制御必須）

---

## 7. Proxy側の制御とUI連携（必須）
- [ ] iOSはAI実行のたびに Proxy から `remaining` を受け取り、UIに反映
- [ ] remaining <= 0 のときはPaywallへ誘導（テンプレ分解へフォールバックも可能）
- [ ] Proxy側で日次上限/分次上限がある（濫用で赤字にならない）
- [ ] clientId をKeychainに保持し、再インストールでも極力同一端末扱い（MVPレベルでも最低限）

---

## 8. Paywall / UX（審査に強い形）
- [ ] Freeでも核機能が使える（テンプレ分解・週次レビューなど）
- [ ] Paywallは「何が増えるか」が明確（AI回数、書き出し、残回数表示）
- [ ] Proの管理（解約/管理画面への導線）がある（Apple標準の管理への誘導でOK）
- [ ] 課金を強要しない（AIがないと全く使えない設計は避ける）

---

## 9. Sandboxテスト観点（最低限）
- [ ] Sandboxユーザーで購入できる
- [ ] サブスク購入→isPro ON→解約/期限切れ（シミュ）→isPro OFF を確認
- [ ] 消耗型を複数回買ってクレジットが加算される
- [ ] Restore Purchases で復元が期待どおり
- [ ] 通信遮断/再起動/アプリ再起動後も状態が崩れない

---

## 10. PRに必ず載せる証拠（Devinへの要求）
- [ ] Paywallスクショ
- [ ] 購入成功後の isPro 表示スクショ
- [ ] pack300購入後のクレジット増加スクショ
- [ ] Restore後に状態が戻るスクショ（またはログ）
- [ ] Proxy remaining の表示スクショ（M11以降）

---

## 11. よくある“地雷”チェック（これが出たら差し戻し）
- [ ] Pro判定がローカルフラグだけ（StoreKitを見てない）
- [ ] 消耗型購入で二重加算が起きる（同じTransactionを何度も処理）
- [ ] remaining計算が端末だけで完結（Proxy制御なし）
- [ ] userCancelled をエラー扱いでアラート連発
- [ ] オフラインでクラッシュ/無限スピナー
